## 1. 产品定位

“向晴简历”是一款面向年轻求职者的 AI 简历优化产品。核心用户包括：

- 应届生、实习生、毕业两年内参加校招的用户；
- 工作约 1—3 年、准备社招或转岗的职场人；
- 因投递无回复而焦虑，希望快速获得专业简历表达的用户。

产品不是简单替换关键词，也不能编造经历。它需要先理解用户做过什么、具备什么能力，再根据目标岗位 JD 调整信息重点与专业表达。

视觉和情绪目标：温暖、年轻、时尚、可信，降低用户在求职过程中的慌张和挫败感。背景使用真实花草与微风动画，保持治愈感但不干扰表单阅读。

## 2. 完整用户流程

1. 用户进入首页，选择“校招 / 应届生”或“社招 / 职场人”。
2. 上传已有简历，支持 DOCX、PDF、TXT；老式 DOC 需要后端转换后解析。
3. 系统读取原简历，自动提取姓名、年龄、手机号、邮箱、教育、工作和项目经历。
4. 用户检查自动识别信息；有误时可手动修改，无误直接继续。
5. 用户填写目标职位、目标行业并粘贴完整岗位 JD。
6. 系统显示分阶段加载状态：解析简历、理解能力、分析 JD、专业重写、真实性检查。
7. 生成两页左右的可用简历，并显示匹配度、关键词覆盖和优化说明。
8. 用户可以免费预览、编辑、复制、查看修改颜色、阅读每处修改原因、比较原版与优化版。
9. 用户点击下载时必须登录；每个唯一账号每天获得 3 次免费下载额度。
10. 当天免费额度用完后才出现付费页面，退出再登录不会重置。
11. 用户选择套餐和微信/支付宝，完成支付后，系统确认到账并向当前唯一用户发放对应权益。
12. 用户下载 Word 文件，并可在 Word 中继续修改。

## 3. 校招与社招必须采用不同优化策略

### 3.1 校招 / 应届生

- 允许较大幅度重构模块顺序和表达方式；
- 深挖重要实习、校园项目、竞赛、课程项目与社团经历；
- 强调成长潜力、学习能力、方法、角色和可验证结果；
- 可以合并弱经历、前置强项目、重写个人优势；
- 不得编造项目、公司、角色、数据或证书。

### 3.2 社招 / 职场人

- 采用“微雕”而非全面重写；
- 公司、岗位、工作时间、业务事实和职业轨迹必须保持不变；
- 从真实工作中选择与 JD 更契合的部分，提高信息优先级；
- 将泛泛职责改成“动作 + 方法 + 结果”，但不能无依据创造数字；
- 保持原履历主干，只调整措辞、顺序和强调点，即“换表达，不换经历”。

## 4. 简历解析与信息确认

### 4.1 文件支持

- TXT：浏览器可直接读取；
- DOCX：当前前端使用 Mammoth 提取真实文字；
- PDF：当前前端使用 PDF.js 逐页提取真实文字；
- DOC：浏览器无法可靠解析，正式后端需调用 LibreOffice 或商业文档转换服务转为 DOCX/PDF；
- 文件大小上限当前为 10MB。

### 4.2 自动识别字段

- 姓名、性别、年龄；
- 手机号、邮箱；
- 教育经历；
- 工作经历、实习经历；
- 项目经历；
- 技能、语言、证书；
- 时间区间、公司、学校、岗位名称。

识别结果必须由用户确认。不能识别时应显示“请确认”，不能用虚构姓名或经历填充。

## 5. AI 优化结果要求

- 正式结果建议 1.5—2 页，本产品默认导出 2 页 A4 Word；
- 内容必须来自用户上传的原始简历；
- 不编造数字、职责、公司、学校或项目；
- 输出包括个人优势、核心能力、工作/实习、重点项目、教育、技能等；
- 对 JD 关键词进行匹配，但不能机械堆词；
- Word 下载时移除网页中的颜色和解释批注，得到干净可编辑版本。

### 5.1 修改可视化

- 黄色：基于原经历进行专业重写；
- 绿色：从原经历归纳、补充能力总结；
- 蓝色：强化 JD 相关关键词；
- 每处彩色改动下方必须显示小字原因，例如：
  - “为了提高与 JD 中跨团队协作要求的契合度”；
  - “从职责描述改为动作、方法和结果，突出个人贡献”；
  - “补全研究方法和产出物，使项目更具体可信”。

### 5.2 原版与优化版对比

- 必须左右横向对比；
- 左侧必须来自用户真实上传的原简历，禁止使用虚构示例代替；
- 右侧显示优化结果和彩色修改范围；
- TXT、DOCX、PDF 应显示真实提取文字；PDF 可额外打开原始排版；
- 对比页面在窄屏可横向滚动，不能自动变成含义不清的上下对比。

## 7. 唯一账号与登录

付费、订单和额度必须绑定唯一用户 ID，不能只认浏览器或文件。

### 7.1 登录方式

- Firebase 邮箱 + 密码（当前已接入，作为首发主登录方式）；
- 手机号 + 短信验证码；
- 微信快捷登录；
- Apple ID 登录。

### 7.2 账号规则

- Firebase 邮箱账号使用不可变 `uid` 映射唯一 `user_id`；
- 手机号首次验证后建立唯一 `user_id`；
- 微信登录使用微信 OpenID/UnionID 映射唯一用户；
- Apple 使用 `sub` 字段映射唯一用户；
- 同一个第三方身份重复登录必须回到同一个账号；
- 换设备登录后，免费额度、订单和会员仍应同步；
- 后续应提供账号绑定/合并能力，避免手机号、微信和 Apple 分裂成三个账号；
- 订单必须记录 `user_id`，支付成功只给该用户发放权益。

当前线上前端已接入 Firebase 邮箱/密码认证，注册用户可在 Firebase Authentication → Users 中查看。微信、Apple 和手机号登录尚未启用。Firebase Authentication 只解决身份唯一性；免费额度、订单、会员和兑换码仍必须迁移到生产数据库或受信任的服务端，不能以浏览器 `localStorage` 作为最终依据。

## 8. 支付闭环

### 8.1 订单链路

1. 已登录用户选择套餐和支付方式；
2. 前端只提交 `plan` 与 `method`，金额由服务端套餐表决定；
3. 服务端创建唯一订单号，例如 `XQ...`，订单状态为 `PENDING`；
4. 服务端调用微信 Native 支付或支付宝当面付，获得真实支付二维码；
5. 用户扫码付款；
6. 微信/支付宝服务器异步通知本系统；
7. 本系统验证签名、商户号、订单号、金额和支付状态；
8. 将订单更新为 `PAID`；
9. 在同一数据库事务中发放单次额度或会员；
10. 将订单更新为 `FULFILLED`；
11. 前端轮询订单状态，显示“已到账 / 已发放”并允许下载。

### 8.2 订单状态

- `PENDING`：已创建，等待付款；
- `PAID`：支付平台确认到账；
- `FULFILLED`：权益已成功发放；
- 后续可增加 `CLOSED`、`REFUNDING`、`REFUNDED`。

### 8.3 到账与发放校验

- 不能因为用户点击“我已支付”就发放权益；
- 只能相信微信/支付宝服务端回调或主动查单结果；
- 回调必须验签；
- 回调金额必须和服务端订单金额完全一致；
- 订单的 `user_id` 决定权益发给谁；
- `provider_transaction_id` 必须唯一；
- 发放逻辑必须幂等，同一回调重复 10 次也只能发放一次；
- 前端成功页显示订单号、到账状态和发放权益；
- 账号中心可以查询最近订单。

当前前端包含“模拟支付成功”按钮，仅用于演示，不产生扣款。`server.py` 的 DEMO 模式也只用于本地联调。

### 8.4 闲鱼首发收款与账号直充

普通个人卖家首发阶段采用：网站为已登录 Firebase 用户创建唯一订单 → 买家在闲鱼付款并通过闲鱼聊天发送网站订单号 → 卖家核对网站订单号、闲鱼订单号和实付金额 → 卖家后台确认 → Firestore 事务标记到账并向该 Firebase UID 发放额度/会员 → 买家页面轮询并显示“已到账 / 已发放”。同一闲鱼订单号只能绑定一个网站订单，重复确认不得重复增加权益。

没有官方卖家回调时由卖家人工确认到账；不得通过爬虫、Cookie 或非授权接口读取闲鱼订单。人工只负责判断闲鱼是否真的到账，金额校验、订单幂等和权益发放由服务端完成。前端默认关闭模拟微信/支付宝入口。通过 Vercel 环境变量配置 `XIANYU_SINGLE_URL`、`XIANYU_BASIC_URL`、`XIANYU_PRO_URL`、`ADMIN_SECRET` 和 Firebase Admin 服务账号。

## 10. 后端接口

当前本地后端文件：[server.py](./server.py)

| 方法 | 路径 | 用途 |
|---|---|---|
| GET | `/api/config` | 返回线上认证能力、支付模式与闲鱼商品链接（不返回密钥） |
| POST | `/api/auth-session` | 验证 Firebase ID Token，并以 Firebase UID 创建/恢复唯一用户 |
| POST | `/api/auth-email` | 同域代理邮箱注册、登录、密码重置和刷新令牌，避免国内浏览器直连 Google 超时 |
| GET | `/api/me` | 获取用户、免费额度、会员和订单 |
| POST | `/api/orders` | 按服务端固定价格创建绑定当前 Firebase UID 的闲鱼待付款订单 |
| GET | `/api/orders?order_no=...` | 仅允许订单所属账号查询订单状态 |
| POST | `/api/admin-confirm` | 卖家核对闲鱼到账后，校验金额与订单唯一性并在事务中发放权益 |
| POST | `/api/quota-consume` | 导出前原子校验并扣减每日免费、单次或会员周额度 |

正式下载建议增加：服务端先调用 `/api/quota/consume`，成功后返回短时有效、只可使用一次的下载令牌，再由下载接口生成 Word。不能让未授权用户直接调用生成接口。

## 11. 核心数据模型

### users

- `uid`：Firebase Authentication 的不可变唯一用户 ID；
- `email`：用户登录邮箱；
- `provider`：当前首发为 `firebase`；
- `free_credits`：初始值 3；
- `created_at`。

### orders

- `order_no`；
- `user_id`；
- `plan`；
- `method`；
- `amount`；
- `status`；
- `provider_transaction_id`；
- `created_at / paid_at / fulfilled_at`。

线上 Firestore 订单另存 `xianyuOrderNo`，并用 `xianyuOrders` 唯一占用记录防止同一闲鱼订单给不同账号重复入账。

### entitlements

- `user_id`；
- `plan`；
- `credits`：单次额度；
- `expires_at`：会员到期时间；
- `usage_date / usage_count`：19.8 会员当前自然周起始日期与本周使用次数。

## 12. 安全与风控要求

- 验证码 5 分钟过期，生产数据库应只存哈希；
- 单手机号、IP、设备需做短信频率限制和图形验证码策略；
- 登录 Session/JWT 应可撤销并设置过期；
- 简历文件属于敏感个人信息，应加密存储并采用短期访问 URL；
- AI 处理前需明确隐私用途，不得用用户简历训练或公开展示；
- 支付金额、权益内容和订单归属必须由服务端判断；

## 19. AI 处理流程与数据契约

### 19.1 处理阶段

1. 文件解析为保留段落顺序的标准文本；
2. 为每个原始片段生成不可变 `source_id`；
3. 结构化抽取个人信息、组织、职位、日期、项目、动作、方法和结果；
4. 用户确认高风险字段；
5. 解析 JD 的职责、硬技能、软技能、行业词和优先级；
6. 根据校招/社招策略选择内容，不直接生成无证据事实；
7. 生成优化文案和每处修改理由；
8. 事实一致性检查；
9. 长度、排版与 ATS 兼容性检查；
10. 保存提示词版本、模型版本和结果版本，便于追溯。

### 19.2 AI 请求输入

```json
{
  "user_id": "usr_xxx",
  "resume_id": "res_xxx",
  "resume_version": 1,
  "applicant_type": "graduate|experienced",
  "target_position": "产品经理",
  "target_industry": "互联网",
  "jd_text": "...",
  "confirmed_profile": {
    "name": "...",
    "age": 24,
    "phone": "...",
    "email": "..."
  },
  "source_fragments": [
    {"source_id": "src_001", "section": "experience", "text": "原始内容"}
  ]
}
```

    }

      "original_text": "...",

### 19.4 强制约束

- `changes[].source_ids` 不能为空，纯版式变化除外；
- 新数字只能来自原简历，禁止模型估算；
- 社招模式下，公司、职位、开始/结束时间必须逐项比对；
- “个人优势”可以归纳，但每个能力声明必须有来源片段；
- 缺失的信息以“待补充”提示用户，不能自动生成；
- 匹配分数必须由已公开的计算规则生成，不能让模型随意给分；
- 模型输出先经过 JSON Schema 校验，再经过事实检查器，最后进入页面；
- 高风险结果必须阻断导出并要求用户确认。

### 19.5 匹配度计算建议

匹配度仅作为表达覆盖参考，不代表获得面试概率。建议构成：

- JD 硬技能覆盖 35%；
- 核心职责证据覆盖 30%；
- 项目/成果证据完整度 20%；
- 简历结构和可读性 10%；
- 行业与岗位基础信息 5%。

页面必须展示计算说明，禁止使用“保证面试”“通过率提升”等无法证明的承诺。

- 任一步可进入 `FAILED`，保留可重试错误码；
- 用户主动取消进入 `CANCELLED`，未成功时不扣额度。

## 23. 账号绑定、合并与注销

### 23.1 绑定

- 已登录用户可以绑定手机号、微信和 Apple；
- 绑定前必须验证新身份；
- 一个第三方身份只能绑定一个主账号；
- 至少保留一种可登录方式，最后一种不能直接解绑。

### 23.2 合并

- 发现新身份已属于另一账号时，不自动覆盖；
- 要求同时验证两个账号后再合并；
- 订单、会员、单次额度、简历版本迁移到主账号；
- 合并过程写入审计日志且不可静默执行。

### 23.3 注销

- 用户需再次验证身份；
- 页面明确说明剩余权益、简历和账号资料将被删除；
- 设置合理冷静期并允许撤回，具体时长由法务确认；
- 冷静期结束后删除简历原件、解析文本、结果和登录身份；
- 支付和财务记录按适用法规保留最小必要信息，并与可直接识别的简历内容分离。

## 24. 订单、售后与退款规则

- 未支付订单 15 分钟后关闭；

- 支付成功但权益发放失败必须自动重试并报警，不能要求用户再次付款；
- 单次额度未使用、会员未发生任何 AI/下载行为时，可提交退款申请；
- 已消费数字权益、部分使用会员、活动赠送权益的退款规则必须在支付前明确展示；
- 重复扣款、金额不符或系统长期不可用优先全额退款；
- 退款成功后回收未使用权益；已使用权益的处理须经客服审核；
- 微信/支付宝退款也必须通过官方接口并保存退款单号；
- 最终退款政策上线前必须由法务审核，页面不得自行承诺与实际能力不符的“无条件退款”。

- 无证据内容拦截率和用户撤销修改率。

### 26.2 事件

`landing_view`、`applicant_type_selected`、`resume_upload_started`、`resume_parse_succeeded`、`profile_confirmed`、`jd_submitted`、`optimization_started`、`optimization_succeeded`、`comparison_opened`、`change_reason_viewed`、`download_clicked`、`login_completed`、`free_credit_consumed`、`paywall_viewed`、`order_created`、`payment_confirmed`、`entitlement_fulfilled`、`docx_downloaded`、`refund_requested`。

埋点不得上传完整简历、JD、手机号、邮箱或支付密钥。用户 ID 使用内部匿名标识。

## 27. 非功能要求

### 27.1 性能目标

- 首页首屏在常见 4G 网络下 LCP ≤ 2.5 秒；
- 文件上传进度必须可见；
- 文字型 PDF/DOCX 解析 P95 ≤ 15 秒；
- AI 优化 P95 ≤ 60 秒，超过 10 秒必须持续反馈阶段；
- DOCX 导出 P95 ≤ 10 秒；
- 支付回调到账到权益发放 P95 ≤ 5 秒；
- 订单查询接口 P95 ≤ 500ms。

### 27.2 可用性与恢复

- 核心 API 月可用性目标 99.9%；
- 支付回调和权益发放任务可重试；
- 生产数据库每日备份，按业务风险配置时间点恢复；
- AI、短信、支付服务异常时应降级并保留用户已填写内容；
- 前端表单自动保存，但敏感原文不长期写入浏览器存储。

### 27.3 兼容与无障碍

- 支持最近两个主要版本的 Chrome、Safari、Edge 和微信内置浏览器；
- 桌面宽度 1280+、平板、手机 375px 均能完成核心流程；
- 所有表单有明确 label、错误提示和键盘焦点；
- 颜色说明同时配文字/图标，不能只靠颜色；
- 正文对比度满足 WCAG AA 目标；
- 尊重 `prefers-reduced-motion`；
- 支付和登录弹窗支持 Esc、焦点圈定和读屏标题。

## 28. 数据生命周期与隐私

- 上传前明确告知用途、保存时间和删除方式；
- 原始文件默认保留 7 天，用于用户重试和对比；用户可随时立即删除；
- 结构化简历与优化结果默认保留 30 天；注册用户可选择长期保存；
- 下载临时文件和令牌最长保留 10 分钟；
- 服务日志不得记录简历正文、验证码、私钥和完整手机号；
- 对象存储使用私有桶、服务端加密和短期签名 URL；
- 数据库敏感字段加密，传输全程 HTTPS；
- AI 服务商必须支持不用于训练/零数据保留或满足产品隐私承诺；
- 用户删除后同步删除对象存储、搜索索引、缓存和 AI 中间产物；
- 支付记录的法定保存期限由财务与法务确认，不能跟随简历一起随意删除。

## 29. 测试计划

### 29.1 必测主路径

- 校招：DOCX → 自动识别 → 修正 → JD → 优化 → 对比 → 登录 → 免费下载；
- 社招：PDF → 事实确认 → JD → 微雕 → 检查公司/日期不变 → 下载；
- 同一账号当天三次免费额度依次扣减，退出重登不恢复，次日自动重置为 3；
- 三种套餐分别下单、到账、发放和下载；
- 手机、微信、Apple 三种账号登录后恢复相同账号权益；
- 支付回调重复、乱序、超时和金额不一致；
- 下载失败重试不重复扣额度。

### 29.2 AI 质量测试集

- 至少 100 份脱敏校招简历和 100 份脱敏社招简历；
- 覆盖互联网、快消、新能源、咨询、制造和职能岗位；
- 对组织、职位、日期和数字做自动一致性比对；
- 人工评分维度：真实性、JD 相关性、清晰度、专业度、可面试解释性；
- 社招事实字段准确率目标 100%；
- 无来源数字进入最终结果的容忍度为 0；
- 每次模型或提示词升级必须跑回归集并保留报告。

### 29.3 支付验收

- 使用微信/支付宝沙箱或最小金额真实环境联调；
- 验证签名失败、错误商户号、错误金额、重复交易号全部拒绝；
- 对账总额 = 成功订单总额 = 支付平台交易总额；
- 权益发放记录与成功订单一一对应。

## 34. 关键术语

- **优化次数**：一次成功返回新 AI 优化结果的任务；
- **下载额度**：成功创建并开始下载一个新 DOCX 的权利；
- **证据映射**：优化内容与原简历 `source_id` 的关联；
- **微雕**：不改变履历事实，只调整措辞、顺序和重点；
- **重构**：在不编造事实的前提下重新组织校招经历；
- **到账**：支付平台服务端确认交易成功；
- **发放**：系统把购买的额度/会员写入指定用户账号；
- **完成订单**：订单状态为 `FULFILLED`，到账且权益已发放。
